# 运行日志检索设计

## 目标

为管理员补齐“程序后台打印日志”的检索能力，让失败请求可以通过请求 ID 或关键词在管理后台直接定位到对应运行日志，而不是只能登录服务器翻文件。

## 现状

- 数据库业务日志已经支持按 `request_id` 查询，但它们不是程序运行日志。
- 程序运行日志由 `logger/logger.go` 输出到 `oneapi-*.log` 文件，日志行已包含请求 ID。
- 新版与旧版后台目前都只有日志文件管理能力，没有运行日志内容检索。

## 方案

第一版直接基于现有日志文件检索，不新增数据库表，也不接入外部日志平台。

### 后端

- 在 root 权限下新增 `GET /api/performance/logs/search`。
- 查询参数：
  - `request_id`
  - `keyword`
  - `limit`（默认 200，最大 1000）
- 必须至少提供 `request_id` 或 `keyword` 之一。
- 搜索范围为当前日志目录中仍保留的全部 `oneapi-*.log` 文件。
- 文件按新到旧扫描；单文件内结果按新到旧返回。
- 返回结构化结果：时间、级别、请求 ID、消息、原始行、来源文件。

### 前端

- 新版前端：在“系统设置 → 运维 → 性能 → 服务器日志管理”下新增“运行日志检索”。
- 旧版前端：在对应“服务器日志管理”区域新增同等能力。
- 两套前端都提供：请求 ID 输入、关键词输入、查询按钮、结果列表、空状态和错误提示。

## 约束与边界

- 只检索仍然保留在日志目录中的文件，已清理文件无法查询。
- 第一版只返回命中的日志行，不做上下文前后行展开。
- 不允许空条件全量扫描。
- 运行日志可能包含敏感信息，因此沿用现有服务器日志管理的 root 权限边界。

## 验证

- 后端单元测试覆盖：条件校验、最新优先、请求 ID 与关键词交集、结果上限、日志行解析。
- 新旧前端分别执行构建或类型检查，确认无编译问题。
